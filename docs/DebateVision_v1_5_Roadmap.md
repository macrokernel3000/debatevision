# DebateVision v1.5 Roadmap

這份文件用來把 DebateVision 從「可用的抽卡網站」整理成「可長期上課的活動引擎」。

## v1.5 目標

讓老師能穩定開課，而不是只讓網站能運作。

每個玩法都應該清楚回答四件事：

1. 開始：這局開始前要設定什麼？
2. 進行：學生現在要做什麼？
3. 結束：怎麼公布、比較、投票或收束？
4. 再來一場：如何快速重開，不需要重整網頁？

## 目前狀態

### 已經穩定的部分

- 卡牌資料集中在 `data/cards/*.csv`。
- 玩法文案集中在 `data/content/玩法文案.csv`。
- 玩法規則集中在 `data/modes/*.json`。
- 大卡圖、抽選池小符號、場景背景已經可以分開管理。
- `data/generated/` 可以由 `網站更新.command` 重建。

### 需要繼續整理的部分

- `website/js/app.js` 已超過 1000 行，玩法邏輯、畫面渲染、編輯模式、抽選池控制都混在同一檔。
- 常用通用介面文字已開始集中到 `data/content/介面文字.csv`。
- 不是所有玩法都有完整的「再來一場」狀態。
- 老師備課流程目前主要靠 README，還缺一份真正的使用手冊。

## 下一階段工作順序

### 1. 玩法生命週期標準化

每個玩法都要有：

- `setup`：本局設定。
- `active`：活動進行。
- `result`：公布結果或產生卡牌後。
- `restart`：再來一場。

這些文字先放在：

```text
website/js/mode-lifecycle.js
```

之後若要更進一步，可移到 CSV 或 JSON。

### 2. 文字模組化

目前已可從 CSV 改：

- 玩法描述
- 教練提示
- 回合流程
- 卡牌任務
- 玩法背景

下一步應考慮移到 CSV 的文字：

- 首頁標題與副標
- 區塊標題，例如「本輪卡牌」「教練提示」「本局抽選池」
- 空狀態文字
- 通用按鈕文字
- 編輯模式文字，例如「圖片微調器」「匯出 JSON」
- 推理解密狀態文字，例如「請選定秘密答案」「再來一場」

建議新增：

```text
data/content/介面文字.csv
```

但不要急著一次搬完，先從老師最常想改的文字開始。

第一批已搬移：

- `本輪卡牌`
- `教練提示`
- `回合流程`
- `本局抽選池`
- `全選目前牌組`
- `取消目前牌組`
- `重置本玩法`
- `再來一場`
- `直接公布答案`

下一批建議搬移：

- 首頁標題與副標
- 圖片微調器文字
- 抽卡機圖片、活動大圖等編輯模式標籤
- 抽選池數量句，例如「固定抽 1 張」「張可抽」

### 3. 圖示三層規則固定

卡牌 CSV 裡的圖示分工：

- `卡牌圖示`：抽出卡牌的大圖示或圖片 ID。
- `抽選池圖示`：底下勾選池的小符號，可填文字、符號或 emoji。
- `圖片`：橫向大圖或特殊卡圖。

這個規則已經進入規格，未來不要再混用。

### 4. 程式拆檔

建議拆分順序：

1. `mode-lifecycle.js`：玩法生命週期文字。
2. `card-renderers.js`：卡牌、token、抽卡機畫面。
3. `pool-selection.js`：抽選池勾選、稀有度篩選、已抽排除。
4. `mode-handlers/`：各玩法自己的開始、抽選、公布、重開邏輯。

拆檔時要保持每一步都能正常運作，不做一次性大搬家。

## 暫時不要做

- 不急著做登入後台。
- 不急著做資料庫。
- 不急著把所有文字一次搬到 CSV。
- 不急著刪掉舊圖片或舊符號，只要規則清楚即可。

## v1.5 完成標準

- 老師看手冊能知道五種玩法怎麼開。
- 每種玩法都能快速開始、進行、結束、再來一場。
- 新增卡牌、圖示、玩法文字不需要改程式。
- `app.js` 開始縮小，至少有第一批獨立模組。
- GitHub Pages 上傳流程不容易漏檔。
